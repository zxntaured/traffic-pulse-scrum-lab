# TrafficPulse

**TrafficPulse** – это консольное приложение для анализа ежедневных маршрутов. Оно помогает пользователю выбрать оптимальное время для поездки, визуализируя гипотетическую или реальную загруженность на пути.

Проект использует Python и фейковые/открытые API для симуляции данных. Цель – не создать продакшен-решение, а отработать процесс командной разработки по методологии Scrum с использованием GitHub Issues.

Ключевые возможности, которые будут реализованы: парсинг входных данных, расчет маршрута, генерация "пробок", ASCII-визуализация, рекомендательная система и экспорт отчета.

## Sprint 1 Review

**Цель спринта:** Реализовать базовый конвейер приложения: ввод данных, расчет и начало визуализации.

**Завершено:**
- ✅ (#1) Реализован парсер для ввода адресов
- ✅ (#2) Добавлен модуль расчета базового времени и расстояния
- ✅ (#8) Создана документация по запуску

**Доставлено:** Пользователь может запустить программу, ввести два пункта и получить расчетное время в пути. Проект готов к интеграции визуализации.

**Не завершено:**
- ❌ (#3) ASCII-визуализация графика пробок. Функциональность реализована наполовину, требует доработки форматирования вывода.

## Sprint 1 Retrospective

**Что прошло хорошо:**
1. Четкое разделение задач по Issue позволило параллельно работать над парсером и калькулятором
2. Раннее создание документации (#8) помогло новым участникам быстро подключиться к проекту
3. Использование меток для приоритетов помогло фокусироваться на важном

**Что пошло не так:**
1. Задача #3 (ASCII-график) оказалась сложнее, чем оценили. Не хватило времени на отладку форматирования вывода
2. Не было ежедневных стендапов (в модели), из-за чего проблема с #3 была обнаружена слишком поздно
3. Переоценили свои силы, взяв в спринт 5 задач вместо рекомендуемых 4

**Что улучшить в следующем спринте:**
1. Проводить более детальный анализ сложности задач перед планированием спринта
2. Обязательно модерировать ежедневное обновление статусов в Issues (через комментарии)
3. Взять в спринт на одну задачу меньше для создания "буфера" на непредвиденные сложности
4. Ввести правило: если задача не сдвинулась за 2 дня, проводить внеплановое обсуждение
